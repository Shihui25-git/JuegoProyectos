01. ORIENTACIÓN DE MISIÓN

ECO_PROJECT es un despliegue operativo diseñado para la interceptación y neutralización de factores de riesgo ambiental. A través de una serie de micro-intervenciones tácticas, el objetivo es consolidar los 17 Objetivos de Desarrollo Sostenible (ODS).

02. PROTOCOLOS DE OPERACIÓN

  • Bucle Crítico: Secuencia de retos rápidos (8-15s) con incremento de entropía (velocidad).
  • Supervivencia: 4 núcleos de integridad disponibles. El agotamiento de núcleos detiene la misión.
  • Progresión: La complejidad aumenta dinámicamente según la efectividad operativa (puntuación).

04. PROBLEMAS DEL PROYECTO

Problema 1: Falta de revisión obligatoria en los Pull Requests

Al inicio del desarrollo del proyecto se utilizó GitHub como plataforma de control de versiones y trabajo colaborativo. Sin embargo, no se configuró desde el principio la opción de protección de ramas que obliga a que los pull requests sean revisados y aprobados por otro miembro del equipo antes de fusionarse con la rama principal.

Debido a esta falta de configuración, cualquier miembro del equipo podía aceptar su propio pull request o fusionarlo directamente sin que otro desarrollador revisara previamente el código. Esto eliminaba una capa importante de control de calidad, ya que no existía una revisión externa que pudiera detectar posibles errores o problemas en la estructura del proyecto.

Posteriormente se identificó este problema y se activó la revisión obligatoria de pull requests, con el objetivo de mejorar el flujo de trabajo y evitar errores en futuras integraciones de código.

Problema 2: Error en la estructura de carpetas del proyecto

Como consecuencia de no contar inicialmente con una revisión obligatoria de pull requests, se produjo un error en la estructura de carpetas del proyecto. Durante una subida de cambios al repositorio, se creó accidentalmente una carpeta adicional que no formaba parte de la organización prevista del proyecto.

Dentro de esta carpeta se generaron otras subcarpetas, provocando una estructura incorrecta y desorganizada del código. Este problema no fue detectado en el momento de realizar la subida debido a que el pull request no fue revisado por otro miembro del equipo antes de integrarse en la rama principal.

Para solucionarlo, se revisó manualmente la estructura del proyecto. El contenido de las carpetas incorrectas fue trasladado a las ubicaciones correspondientes dentro del proyecto, reorganizando correctamente los archivos. Finalmente, la carpeta adicional fue eliminada para restaurar la estructura adecuada del repositorio.

05. LECCIONES APRENDIDAS

Durante el desarrollo del proyecto se comprendió la importancia de configurar correctamente las herramientas de trabajo colaborativo desde el inicio. El uso de plataformas como GitHub no solo permite almacenar el código, sino también establecer mecanismos de control que ayudan a mantener la calidad del proyecto.

Uno de los principales aprendizajes fue que la revisión de código entre miembros del equipo es fundamental para detectar errores antes de que se integren en la rama principal. La ausencia de este proceso puede provocar problemas en la organización del proyecto o en el funcionamiento del código.

Además, se reforzó la importancia de mantener una estructura de carpetas clara y revisada antes de subir cambios al repositorio, ya que pequeños errores al organizar archivos pueden generar confusión o desorden en el proyecto.

06. MEDIDAS ADOPTADAS PARA EVITAR LOS MISMOS ERRORES

Para evitar que estos problemas vuelvan a producirse, el equipo adoptó varias medidas dentro del flujo de trabajo del proyecto.

  • En primer lugar, se configuró la protección de la rama principal en GitHub para que todos los cambios deban realizarse mediante pull requests que requieran la revisión y aprobación de al menos otro miembro del equipo antes de poder fusionarse.
  • Además, se estableció como práctica revisar la estructura de carpetas y los archivos incluidos antes de realizar cualquier subida al repositorio. De esta forma se reduce la probabilidad de subir archivos o carpetas innecesarias.
  • Por último, se acordó mantener una comunicación más activa entre los miembros del equipo durante el proceso de desarrollo, con el objetivo de detectar posibles errores lo antes posible y mantener una organización clara del proyecto.
